- 测试配置：B=1, S=16384, SKV=131072, H=128, HKV=1, DQKV=576, DV=512, topk=2048

## 一、preprocess_delta_kernel\<576\>

### 1.1 基本指标

| 指标 | 数值 |
|------|------|
| 耗时 | 728.26 µs |
| Grid Size | (16384, 8, 1)，共 131072 blocks |
| Block Size | (128, 1, 1) |
| 计算吞吐 (SM Throughput) | 35.81% |
| 访存吞吐 (Memory Throughput) | 5.92 TB/s |
| Achieved Occupancy | 65.59% |
| Theoretical Occupancy | 75.0% |
| Issue Slot 发射间隔 | 2.8 cycles/指令（理想为 1.0） |
| Eligible Warps/Cycle | 0.51（上限 16） |

### 1.2 瓶颈分析

#### 瓶颈一：Memory-Bound（DRAM 为主要限制）

NCU 诊断结论：`SOLBottleneck → DRAM`。Memory Throughput 达到 5.92 TB/s，kernel 的执行时间主要由访存延迟决定，而非计算。

#### 瓶颈二：L1TEX Stall 严重（根因）

- 每个 warp 平均等待 **25.7 cycles** 在 L1TEX scoreboard 依赖上
- 该 stall 占总 stall（29.0 cycles/指令）的 **88.8%**
- 直接导致 Issue Slot 利用率极低，大量调度周期空转

根因定位：NCU 同时报告 global store 访存不合并——每个 cache sector（32 bytes）只有 **16 bytes 被实际利用（50% 浪费）**，说明线程对 delta 写出时存在 stride，引发 uncoalesced global store，放大了 L1TEX 压力。

#### 瓶颈三：Occupancy 受寄存器数量限制

- 理论 Occupancy 75%，低于硬件上限 100%（每 SM 实际只能调度 12 warps，而非上限 16）
- 限制原因：每线程寄存器用量过多
- Occupancy 不足进一步减少了可用于隐藏延迟的 warp 数量，加剧了 stall 影响

### 1.3 优化方向

| 优先级 | 问题 | 优化方向 | 预期收益 |
|--------|------|---------|---------|
| P0 | Global store 未合并（50% sector 浪费） | 调整线程到输出数据的映射，使相邻线程写连续地址；检查 delta 输出 layout 是否存在 stride | 减少 L1TEX stall，提升有效带宽利用率 |
| P1 | L1TEX stall 占 88.8% | 将频繁读取的中间数据（如 softmax 分母、lse）放入 shared memory，减少对 global/L1 的依赖；考虑对输入使用 `__ldg` 只读缓存 | 降低 stall cycles，提升 Issue Slot 利用率 |
| P2 | 寄存器限制 Occupancy（75% → 理想 100%） | 添加 `__launch_bounds__(128, 16)` 指导编译器控制寄存器分配；或拆分计算减少同时活跃变量数 | 提升 Occupancy，增加延迟隐藏能力 |
| P3 | Issue Slot 利用率低（发射间隔 2.8 cycles） | 在 P0/P1 解决后自然改善；若仍不足可考虑指令重排或循环展开增加 ILP | 接近 1.0 cycles/指令 |

### 1.4 下一步行动

1. **定位 uncoalesced store**：通过 NCU Source Counters 找到具体的写出指令行号，确认 delta 写出时的 stride 模式
2. **验证 shared memory 可行性**：评估 preprocess_delta 的数据复用率，判断 smem 搬运是否值得
3. **重新 profile**：修改后用相同配置重新采集，对比 Memory Throughput、L1TEX stall cycles、Occupancy 三项核心指标

## 二、test_mla_bwd_kernel

### 2.1 基本信息

- 采集文件：`flash_mla_bwd_main_profile.ncu-rep`（`profile_bwd.py` + `cudaProfilerStart/Stop` 精准标记）
- 完整 kernel 名：`void test_mla_bwd_kernel<TmaParams<...SM100_TMA_2SM_LOAD_NOSPLIT...>>(SparseAttnBwdParams, T1)`

### 2.2 基本指标

| 指标 | 数值 |
|------|------|
| NCU 测量耗时（含 replay） | 2.39 s |
| Block Size | 512 |
| Cluster Size | **2**（Blackwell 2-SM CTA 协同模式） |
| Grid Size | 32768 |
| Waves Per SM | 221.41 |
| 每线程寄存器数 | **128** |
| Static Shared Memory | 0 byte/block |
| 计算吞吐 (SM Throughput) | **2.88%** |
| 访存吞吐 (Memory Throughput) | **3.03 GB/s** |
| Mem Busy | 2.92% |
| Achieved Occupancy | **25.0%** |
| Theoretical Occupancy | **25.0%**（寄存器限制） |
| Issued Warp Per Scheduler | **≈ 0.00** |
| Issue Slot 发射间隔 | **256.1 cycles/指令** |
| L1TEX Stall 平均等待 | **637.4 cycles/warp** |

### 2.3 瓶颈分析

NCU 总体诊断：`SOLBottleneck → 计算与访存双低`，compute 和 memory 利用率均远低于 60%，属于严重低效。

#### 瓶颈一：Shared Memory Bank Conflict（最严重）

- **Store**：平均 **5.8-way** bank conflict，307M 次 shared store 请求产生 **14.7 亿次 bank conflict**
- **Load**：平均 **4.4-way** bank conflict，4.5M 次请求产生 1452万次 bank conflict
- NCU `UncoalescedSharedAccess`：**9.19 亿次 excessive wavefronts（占总 wavefronts 的 75%）**
- 这是导致 Stall 极高的根本原因之一

#### 瓶颈二：L1TEX Stall 极度严重（637 cycles/warp）

- 每 warp 平均 stall 637.4 cycles 等待 L1TEX scoreboard 依赖（preprocess_delta 仅 25.7 cycles，此处高出 **25 倍**）
- Issue Slot 利用率极低：每 256.1 cycles 才发射一条指令
- 关联原因：Local Memory 访问利用率仅 **1.6/32 bytes（5%）**，表明存在**寄存器 spill 到 local memory**，形成极碎片化的 L1TEX 访问，大量访存延迟堆叠

#### 瓶颈三：Occupancy 严重受限（25%，寄存器压力）

- 128 registers/thread × 512 threads/block = **65536 registers/block**
- 理论 Occupancy 仅 25%（4 warps/scheduler），完全无延迟隐藏能力
- NCU 确认：`TheoreticalOccupancy limited by the number of required registers`

#### 瓶颈四：L2 负载不均衡

- 最热 L2 Slice 活跃 cycles 比平均高 **91.2%**，最冷比平均低 **81.0%**
- 表明 global 访存地址分布不均，部分 L2 slice 成为热点

#### 瓶颈五：Thread Divergence + FP 指令未融合

- 存在 warp 内线程分叉（控制流不统一）
- 0 个 fused FP32 指令，131072 个 non-fused FP32 指令，未充分利用 FMA 指令

### 2.4 优化方向

| 优先级 | 问题 | 优化方向 | 预期收益 |
|--------|------|---------|---------|
| P0 | Shared Memory Bank Conflict（store 5.8x，75% 冗余 wavefront） | 检查 shared memory layout，调整 padding（如每行 +1 或 +2 元素 padding）消除 bank conflict；检查 smem 写入 stride 是否为 bank 数（32）的整数倍 | 大幅降低 smem stall，提升 issue rate |
| P0 | Local Memory Spill（1.6/32 bytes 利用率） | 减少寄存器压力：拆分大 kernel 为更小的计算单元；重构寄存器复用逻辑；使用 `__launch_bounds__(512, 1)` 限制寄存器上限 | 消除 L1TEX spill stall，降低 637 cycles/warp |
| P1 | 寄存器限制 Occupancy（25%） | 目标降低至 ≤96 registers/thread（可将 Occupancy 提升至 33%）；或 ≤64（50%）；与 P0 协同解决 | 提升延迟隐藏，减少 stall 影响 |
| P2 | L2 负载不均衡（+91% 热点） | 检查 global 访存地址计算，引入地址哈希或调整数据排布使访问均匀分散到所有 L2 slices | 提升 L2 带宽利用率 |
| P3 | Thread Divergence | 检查 warp 内的条件分支，尝试用 predicate 替代 branch | 减少 warp divergence overhead |
| P3 | FP 指令未融合 | 检查 FP32 运算是否为连续 mul+add，确认编译器 FMA 优化未被阻断（注意中间强制转换或 volatile 等） | 提升算术吞吐 |

### 2.5 与 preprocess_delta 的对比

| 指标 | preprocess_delta | test_mla_bwd | 差异 |
|------|-----------------|--------------|------|
| SM Throughput | 35.03% | 2.88% | bwd 低 12x |
| Memory Throughput | 5.93 TB/s | 3.03 GB/s | bwd 低 ~2000x |
| L1TEX Stall | 25.7 cycles | **637.4 cycles** | bwd 高 25x |
| Issue Interval | 2.8 cycles | **256.1 cycles** | bwd 高 91x |
| Occupancy | 64.6% | **25.0%** | bwd 低 2.6x |
| Smem Bank Conflict | 无报告 | **5.8x store** | bwd 问题严重 |

`test_mla_bwd_kernel` 是当前整个反向算子的**绝对性能瓶颈**，smem bank conflict + 寄存器 spill 是两个必须优先解决的问题。
