# EduAgent 实现思路简述

## 1. 项目定位

EduAgent 的实现思路，本质上是把教育场景里高频且可标准化的任务拆成多个独立 Agent，再用统一的前后端框架把它们组织成一个可协同、可追踪、可管理的平台。

## 2. 整体架构思路

项目采用 `React + Express + tRPC + MySQL(Drizzle) + LLM API` 的分层结构：

- 前端负责按场景提供页面入口，例如问答、批改、简历审查、模拟面试、知识库管理和历史记录。
- 后端负责把每个教育能力封装成独立业务接口，并处理鉴权、角色权限、数据存储和文件上传。
- LLM 负责完成自然语言理解、内容生成和评分分析。
- 数据库负责保存对话、批改记录、简历报告、面试记录和知识库内容，保证结果可回看、可审核、可沉淀。

## 3. 核心实现方式

### 3.1 用四类 Agent 对应四类核心教育任务

- `QA Agent`：处理课程知识问答，结合多轮上下文和知识库内容返回答案，并标记置信度。
- `Grading Agent`：将试卷批改拆成客观题、简答题、代码题三类思路，统一输出结构化评分结果。
- `Resume Review Agent`：围绕六个维度评估简历，输出分项分数、修改建议和雷达图数据。
- `Interview Agent`：把模拟面试设计成 `intro -> tech -> project -> report` 的阶段式流程，最后生成完整评估报告。

### 3.2 用 tRPC 把 Agent 能力包装成业务接口

后端没有把模型调用直接暴露给前端，而是先经过 tRPC 路由封装：

- 前端只关心“提交问题/试卷/简历/面试回答”。
- 后端负责调用对应 Agent、解析结构化结果、写入数据库，再把最终结果返回前端。

这样做的好处是，模型能力、业务规则和数据读写被放在服务端统一管理，后期更容易扩展和维护。

### 3.3 用结构化输出降低 LLM 不稳定性

项目里很多 Agent 都不是只返回一段自由文本，而是要求模型按 JSON Schema 输出结果，例如：

- 试卷批改输出题目得分、总分、薄弱点和建议。
- 简历审查输出维度分数、修改建议和总结。
- 面试总结输出综合评分、优势、不足和建议。

这种做法可以让前端更稳定地渲染结果，也方便把数据直接存库和做可视化。

## 4. 数据闭环设计

EduAgent 不只是“调用一次模型”，而是设计了完整的数据闭环：

- 学员发起问答、批改、简历审查、模拟面试。
- 后端把过程和结果写入数据库，形成历史记录。
- 教师可以审核知识库和批改结果。
- 管理员可以管理用户角色和系统配置。
- 低置信度问答会进入待补充队列，反过来驱动知识库迭代。

这意味着系统会随着使用不断积累内容，而不是每次都从零开始回答。

## 5. 知识库与多 Agent 协同

项目的一个关键思路是“先知识库，后大模型；先单 Agent，必要时再多 Agent 协同”：

- 问答场景先做知识库检索，命中后优先返回知识库答案，减少幻觉。
- 没有命中时再回退到 LLM 生成。
- 对复杂请求，系统可以先做意图识别，再决定调用哪个 Agent，或者组合多个 Agent 处理。

这让系统既保留了大模型的灵活性，也尽量增强了答案的可控性和业务适配性。

## 6. 一句话总结

EduAgent 的实现思路可以概括为：以教育任务为中心拆分多 Agent，以 tRPC 统一封装后端能力，以结构化输出保证结果可用，以数据库和知识库形成持续迭代的教学闭环。

## 7. 运作流程图

```mermaid
flowchart TD
    U["用户"] --> FE["前端页面"]
    FE --> API["tRPC 接口层"]
    API --> AUTH{"鉴权与角色校验"}

    AUTH --> ROUTE{"选择业务能力"}
    API -. "复杂请求可选" .-> PIPE["意图识别"]
    PIPE --> ROUTE

    ROUTE --> QA0["课程问答"]
    ROUTE --> GR0["试卷批改"]
    ROUTE --> RE0["简历审查"]
    ROUTE --> IN0["模拟面试"]
    AUTH --> MGMT["教师与管理员管理功能"]

    QA0 --> KB["检索知识库"]
    KB --> KBHIT{"是否命中"}
    KBHIT -->|"是"| QA1["直接返回知识库答案"]
    KBHIT -->|"否"| QA2["调用 QA Agent"]
    QA2 --> CONF{"置信度是否较低"}
    CONF -->|"是"| LOW["进入低置信度队列"]
    CONF -->|"否"| QA3["生成问答结果"]
    QA1 --> SAVEQA["保存对话记录与来源"]
    QA3 --> SAVEQA
    LOW --> KBFB["教师补充或修正知识库"]

    GR0 --> GR1["提交试卷或作业"]
    GR1 --> GR2["调用 Grading Agent"]
    GR2 --> GR3["输出题目得分 薄弱点 总评"]
    GR3 --> SAVEGR["保存批改记录"]
    SAVEGR --> REVIEW["教师审核与发布"]

    RE0 --> RE1["提交简历文本或文件"]
    RE1 --> RE2["调用 Resume Review Agent"]
    RE2 --> RE3["输出六维评分 修改建议 雷达图"]
    RE3 --> SAVERE["保存审查结果"]

    IN0 --> IN1["创建面试会话"]
    IN1 --> IN2["Interview Agent 分阶段对话"]
    IN2 --> IN3["生成面试报告与雷达图"]
    IN3 --> SAVEIN["保存面试记录"]

    MGMT --> KBUP["知识库上传与审核"]
    MGMT --> USERMG["用户角色管理"]
    MGMT --> STATS["系统统计与历史查看"]

    SAVEQA --> DB["MySQL 与 Drizzle"]
    KBFB --> DB
    REVIEW --> DB
    SAVERE --> DB
    SAVEIN --> DB
    KBUP --> DB
    USERMG --> DB
    STATS --> DB

    DB --> OUT["历史记录 报告展示 可视化分析"]
```
